关卡原址

先让我们去看看所谓的相关文件吧

细心的你是否已经发现flag01的权限上有SetUID位，而其拥有者又是flag01。这说明了我们在执行flag01的时候拥有flag01用户的权限。这是一条重要的线索。

UID(User identifier)是啥？

虽然我们平常使用的用户都是字符串形式的，但是Unix系的系统在内部都是通过UID（一个整数）来区分用户的。字符串和实际的UID的映射存储在/etc/passwd中。

在一个进程中，通常包含了三种类型的UID：

  • ruid(Real user ID)
  • euid(Effective user ID)
  • suid(Saved user ID)

而suid主要用在由高权限转换为低权限后再获取回高权限的时候。

参考：

  • wiki
  • Real and Effective IDs

让我们来分析一下代码吧

#include <stdlib.h>
#include <unistd.h>
#include <string.h>
#include <sys/types.h>
#include <stdio.h>

int main(int argc, char **argv, char **envp)
{
  gid_t gid;
  uid_t uid;
  gid = getegid();//获得egid
  uid = geteuid();//获得euid

  //将当前进程的rgid，egid，sgid都设置为之前获得的egid
  setresgid(gid, gid, gid);
  //同理设置UID
  setresuid(uid, uid, uid);
  //通过system来开启新的进程
  system("/usr/bin/env echo and now what?");
}

程序弱点分析

  • 线索1：这是一个拥有setuid位的程序，且拥有者为flag01
  • 线索2：程序将当前进程的resgid和resuid都设置为了当前的有效ID
  • 线索3：调用了system函数来执行命令，由于之前的设置，system开启的新进程rid将会变成之前设置的eid
  • 线索4：使用了/usr/bin/env来执行命令，这个命令会将当前的环境变量代入，参考env的wiki
